# bipline

Turn your GitHub shipping activity into reviewed build-in-public drafts in Buffer.

| Property | Details |
|---|---|
| **Status** | Draft, v0.1 |

---

## 1. Overview

bipline watches for the moments worth talking about in your repos (a release published, a meaningful PR merged) and turns them into draft social posts written in your voice. The drafts land in Buffer as unscheduled drafts. You review, edit, and schedule them inside Buffer. Nothing ever publishes without you.

The whole thing runs as a Go CLI invoked by a GitHub Action. No daemon, no polling loop, no hosting, no database.

> **Why build this:** It solves a real operational pain (the gap between shipping and posting about it), it is a genuine "developer building on Buffer's new API" artifact at a moment when that API is days old, and it doubles as a build-in-public content engine for the exact thing it does. It also shows range in one small project: typed GraphQL client generation, an LLM drafting layer with voice constraints, and CI as an event source.

### The bootstrap loop

The first build-in-public post about bipline should be generated by bipline. Cutting the `v0.1` release of the tool triggers its own Action, which drafts the announcement, which you review in Buffer and post. The tool ships itself, and that is the hook for the first post.

## 2. Goals

### In scope (v0.1 MVP)

- A typed Go client for the Buffer GraphQL API (generated, not hand-rolled), covering: resolve organization, list channels, create draft post.
- A draft generator that takes a GitHub event payload and returns a post draft written in your voice via the Claude API.
- A CLI that reads a GitHub event from `GITHUB_EVENT_PATH`, runs the generator, and creates a Buffer draft.
- A GitHub Actions workflow that fires on `release: published` and runs the CLI.
- Idempotency: re-running the same workflow must not create duplicate drafts.
- Deployed in one real repo first (SafeNudge), then the bipline repo itself.

### Out of scope (v0.1)

- Any custom review UI. Buffer is the review surface.
- Third-party OAuth or multi-tenant support. Single user, personal API key only.
- A long-running daemon or tray app.
- Auto-publishing or auto-scheduling. Drafts only.
- Multi-platform variant generation (X vs LinkedIn phrasing). Single draft for v0.1.
- Image or video assets in posts.
- Scanning historical activity. Forward-looking from install.

## 3. Architecture

A linear pipeline, each stage a separate Go package so it is testable in isolation.

```
GitHub release published
        │
        ▼
GitHub Action  ──>  bipline CLI
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
   event parser    draft generator    buffer client
   (read payload)  (Claude API)       (GraphQL, genqlient)
                                            │
                                            ▼
                                   Buffer draft post
                                   (reviewed by you in Buffer)
```

### Components

- **`eventsource`** reads and normalizes the GitHub event. In CI the payload is a JSON file at `GITHUB_EVENT_PATH`. Normalizes a release (or merged PR) into a small internal struct: title, body, repo, url, tag, notable commits.
- **`generator`** takes the normalized event plus a voice config and calls the Claude API. Returns a draft string. Voice rules are not duplicated here; they reference the same constraints as the `humanizer` and `devgiroux-writing` skills (no em dashes, learner not expert, specific over abstract, natural contractions, English only).
- **`buffer`** the generated typed client. Resolves the org ID, resolves the target channel by service name, creates a draft post.
- **`cmd/bipline`** the CLI that wires the three together and handles config, flags, and exit codes.

## 4. Buffer API integration

Grounded in the live docs as of June 2026. Confirm exact required arguments in the API Explorer during the Phase 0 spike before generating bindings.

- **Endpoint:** `https://api.buffer.com` (single GraphQL endpoint).
- **Auth:** `Authorization: Bearer <BUFFER_API_KEY>` header. You already have the key.
- **Object model:** `Account → Organizations → Channels → Posts`. Ideas belong to an Organization.
- **Resolve org:** `account { organizations { id name } }`.
- **Resolve channels:** `channels(input: { organizationId }) { id name service }`. The `service` field is the platform (`twitter`, `linkedin`, etc.), which is how bipline maps a config like `target: linkedin` to a channel ID.
- **Create draft (primary path):** `createPost` mutation with `saveToDraft: true`. Draft posts have several additional required arguments, so verify the exact shape in the Explorer first. The mutation returns a union, success or error, so the client must handle both branches (`... on PostActionSuccess` / `... on MutationError`).
- **Alternative path to consider in the spike:** `createIdea(input: { organizationId, content: { title, text } })` parks content in the Ideas board, org-level, not yet tied to a channel. Decide in Phase 0 whether a channel-assigned draft or an Idea is the better review bucket for build-in-public. Default assumption: channel draft, since you usually know the platform.

### Codegen approach

Use `genqlient`. The flow:

1. Pull the schema by introspection from `https://api.buffer.com` into `schema.graphql`.
2. Write your operations as `.graphql` files (`GetOrganizations`, `GetChannels`, `CreateDraftPost`).
3. `genqlient.yaml` points at the schema and operations.
4. `go generate` produces typed Go functions and response structs.

This is the "typed client for free" part. You write GraphQL, not Go HTTP plumbing.

## 5. Draft generator

Input: the normalized event struct. Output: a single draft string.

- Model: Claude API (Messages). System prompt carries the voice constraints; user message carries the event context (release notes, PR title and body, repo name, a few notable commit subjects, the link).
- Voice config lives in a small file (`voice.md` or similar) so it is editable without recompiling. Seed it from your existing `devgiroux-writing` and `humanizer` rules so there is one source of truth.
- The generator does not post. It returns text. The CLI decides what to do with it. This keeps the LLM layer pure and testable.
- Golden tests: feed 2 to 3 real past events (a SafeNudge release, a BaseVault PR) and assert the output passes basic voice checks (no em dashes, contains the repo link, under platform length limits). You are not asserting exact wording, just invariants.

## 6. GitHub Action trigger

A workflow file dropped into each repo you want covered.

- Trigger: `on: release: { types: [published] }` for v0.1. A `pull_request: { types: [closed] }` path gated on `merged == true` and a label (e.g. `ship`) is a Phase 6 extension.
- Steps: checkout, set up Go, run the bipline CLI with the event payload, secrets injected as env (`BUFFER_API_KEY`, `ANTHROPIC_API_KEY`).
- Idempotency: write a marker so a re-run does not double-draft. Simplest version, include the release tag or event ID in the draft request and skip if a draft for that ID already exists, or record processed IDs in a tiny committed file or an Action cache. Decide the mechanism in Phase 4.

## 7. Phases

Same rhythm as your other study guides: a short reading and decision pass, then a build pass, then review against acceptance criteria. Do not advance until the criteria pass. Each phase ends with something real committed, so the history reads as genuine incremental work.

| Phase | Goal | Acceptance criteria |
|---|---|---|
| **0. Spike** | De-risk Buffer. Get the key working, explore the Explorer, decide draft-post vs Idea, confirm required args. | You have manually created a draft post (or Idea) in your Buffer account via a raw `curl` to `api.buffer.com`. You can list your orgs and channels. The draft path and its required args are written down. |
| **1. Buffer client (Go)** | The typed client. Independently useful, ships on its own. | `genqlient` generates bindings from the live schema. A small program resolves your org, finds a channel by service, and creates a draft. Both success and `MutationError` branches handled. Committed with a short README. |
| **2. Draft generator (Go)** | Event payload to draft string via Claude. | `generator.Draft(event, voice)` returns a draft. Golden tests on 2 to 3 real past events pass the voice invariants. No network calls to Buffer in this package. |
| **3. Event source (Go)** | Normalize a GitHub event. | Given a sample `release.published` payload (and a sample merged-PR payload), the parser returns the normalized struct. Unit tested against fixture JSON. |
| **4. Glue: CLI + Action** | Wire it together, run in CI. | The CLI reads `GITHUB_EVENT_PATH`, drafts, and creates a Buffer draft. The workflow is in SafeNudge. A test release produces exactly one Buffer draft. A re-run produces zero new drafts (idempotency works). |
| **5. Dogfood** | The tool ships itself. | bipline's own repo has the workflow. Cutting `v0.1` produces a draft announcing bipline. You review in Buffer and publish. That is the first post. |
| **6. Optional** | Reach. | Any of: merged-PR trigger with label gate, multi-repo hub on a cron, per-platform variants (X punchy, LinkedIn reflective), a small React review pane if you decide Buffer's review UX is not enough. |

## 9. Defensibility notes

Things to be able to explain without notes, since this is portfolio-adjacent and the Buffer role would ask:

- **Why codegen over a hand-written client.** Typed responses, the schema is the contract, less plumbing to maintain when Buffer's API changes.
- **The union return pattern.** `createPost` returns success or error as a GraphQL union, and why that is better than HTTP status codes for a mutation that can fail business-logic checks.
- **Why CI as the event source** instead of a daemon. No infra, no auth-token-holding service running somewhere, the event you care about (a release) is already a CI event.
- **The idempotency problem.** Workflows re-run. Without a guard, a re-run double-posts. How you solved it and the tradeoffs of the chosen mechanism.
- **Why drafts, never auto-publish.** The human gate is the point. The tool removes the operational friction, not the judgment.

## 10. Open questions

- **Draft post vs Idea** as the v0.1 review bucket. Resolve in Phase 0 with the real API in front of you.
- **Idempotency mechanism.** Committed file vs Action cache vs checking Buffer for an existing draft with a matching marker. Decide in Phase 4 once you have seen how Buffer represents drafts.
- **Notable-commit selection.** For a release, how do you pick which commits feed the prompt? All since last tag, or only those matching a convention? Affects draft quality. Start simple (release notes only) and revisit.
- **Length and platform.** v0.1 produces one draft. When you add variants, the generator needs per-platform length and tone, which your `devgiroux-writing` skill already encodes.
